# Product Requirements Document: Pet Registration API

## Overview

The Pet Registration API provides a standardized, efficient way for veterinary clinics and hospitals to register new pet patients. When a new animal is admitted or scheduled for care, clinic staff must record its essential identification details and link it to a responsible guardian/tutor.

This feature solves the problem of manual, inconsistent record-keeping by providing a structured endpoint to register pets. It is designed specifically for veterinary professionals to enable rapid intake of animals under their care.

## Goals

- **Accurate Patient Intake**: Capture essential details (name, breed, birth date, and tutor name) for every registered pet.
- **Data Quality**: Ensure date of birth validation to prevent invalid future dates from corrupting patient records.
- **Frictionless Workflow**: Support rapid entry of pet information with minimal validation rules (free-text breeds, duplicates allowed).

## User Stories

### Primary Persona: Clinic Receptionist & Administrator

- **As a** clinic receptionist,
  **I want to** register a new pet with its name, breed, date of birth, and tutor's name,
  **so that** we have an active patient record in our system before the animal receives care.

- **As a** clinic administrator,
  **I want to** be blocked from saving a pet with a future date of birth,
  **so that** we avoid incorrect data entry that could impact medical calculations (like drug dosages based on age).

- **As a** clinic receptionist,
  **I want to** register multiple pets with the same name, breed, and tutor (e.g. siblings from the same litter),
  **so that** I don't get blocked by arbitrary duplication rules during intake.

## Core Features

### 1. Pet Intake Submission
- **Description**: Accepts details for registering a single pet.
- **Requirements**:
  - Requires the pet's name, breed, date of birth, and tutor's name.
  - Returns a unique identifier (`id`) generated by the system for the newly registered pet.
  - Returns all registered fields in the response body.

### 2. Basic Attribute Validation
- **Description**: Validates input data to guarantee integrity.
- **Requirements**:
  - **Pet Name**: Must be a non-empty, trimmed string.
  - **Breed (Raça)**: Must be a non-empty string. System treats this as free text (any string is accepted).
  - **Date of Birth (Data de Nascimento)**: Must be a valid date representation. Must be a date in the past (less than or equal to the current system date). Future dates are rejected.
  - **Tutor Name**: Must be a non-empty, trimmed string representing the primary contact.

## User Experience

1. **Intake Entry**: The clinic user enters the pet's name, breed, birth date, and tutor's name.
2. **Instant Validation**:
   - If any required field is missing, or if the birth date is in the future, the system displays a clear, actionable validation error (e.g. "Birth date cannot be in the future").
3. **Confirmation**:
   - On successful submission, the system displays a success confirmation showing the generated Pet ID and a summary of the registered details.

## High-Level Technical Constraints

- **Date Format**: The birth date must be submitted in standard date format (YYYY-MM-DD).
- **ID Generation**: The system must assign a unique, non-guessable identifier to each pet upon successful registration.
- **Response Protocol**: Standard JSON request/response formats.

## Non-Goals (Out of Scope)

- **Separate Tutor Entities**: Creating or managing separate owner accounts/records is out of scope. The tutor is represented purely as a string field on the pet record.
- **Breed Taxonomy**: We will not validate breeds against a controlled vocabulary or list of officially recognized breeds.
- **Pet Management (CRUD)**: Retrieving, updating, or deleting registered pet profiles is deferred to a future phase.
- **Duplicate Detection**: The system will not warning or block duplicate registrations.

## Phased Rollout Plan

### MVP (Phase 1)
- Register endpoint (`POST /pets` or equivalent) with the four required fields.
- Mandatory past-date validation for the date of birth.
- Free-text breed and simple-string tutor name.
- Success criteria: Successful integration with clinic client apps for basic intake.

### Phase 2 (Future)
- Add retrieval (`GET /pets/{id}`) and update (`PUT/PATCH /pets/{id}`) endpoints.
- Allow optional secondary contacts (phone/email) for tutors.

### Phase 3 (Future)
- Transition to a separate `Tutors` resource to allow linking multiple pets to a single owner profile.

## Success Metrics

- **100% Validation Rate**: All invalid requests (e.g. future birth dates, empty names) are correctly blocked.
- **Registration Success**: 100% of valid payloads are successfully processed and return a 201 Created status code.
- **Frictionless Entry**: Receptionists can complete a pet registration in under 10 seconds in client software.

## Risks and Mitigations

- **Risk: Inconsistent Breed Data**: Free-text breeds will lead to spelling variations (e.g., "Labrador", "labrador retriever", "lab"). This will make searching/filtering by breed difficult in the future.
  - *Mitigation*: We accept the risk for the MVP to prioritize speed. We plan to introduce an autocomplete suggestion or a standardized dropdown selection in Phase 2.
- **Risk: Duplicate Records**: Receptionists might accidentally submit the same pet registration twice.
  - *Mitigation*: The generated ID will distinguish them, and we will rely on manual data cleanup or future deduplication UI controls if duplicate records become an operational issue.
