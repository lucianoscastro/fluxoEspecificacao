# ADR-002: Technical Stack and Design Decisions for Pet Registration API

## Status

Accepted

## Date

2026-06-02

## Context

We need to implement the pet registration feature. The project is currently a blank ASP.NET Core minimal Web API. We need to decide on data persistence, code structure, and the testing strategy.

## Decision

We have decided on the following technical design choices:
1. **In-Memory Storage**: Store pet data in an in-memory thread-safe collection (`ConcurrentDictionary`) registered as a singleton service.
2. **Single-File Architecture**: Consolidate models, handlers, and routes directly inside `Program.cs`.
3. **HTTP Integration Testing**: Use `WebApplicationFactory` from `Microsoft.AspNetCore.Mvc.Testing` to perform E2E/integration tests in the test project `fluxoEspecificacao.Tests`.
4. **RFC 7807 Error Responses**: Use the built-in `Results.ValidationProblem` to return standard problem details for invalid payloads.

## Alternatives Considered

### Alternative 1: EF Core with SQLite

- **Description**: Configure EF Core and use a local SQLite database for file-based persistence.
- **Pros**: Persistent data that survives app restarts.
- **Cons**: Adds overhead of installing NuGet packages, configuring DbContext, managing migrations, and testing database setup.
- **Why rejected**: In-memory storage is sufficient for prototyping the initial MVP registration endpoint, avoiding unnecessary configuration.

### Alternative 2: Multi-File Directory Structure

- **Description**: Organize codebase into folder structures like `Models/`, `Handlers/`, or `Endpoints/`.
- **Pros**: Clearer separation of concerns as the app grows.
- **Cons**: Over-engineering for a single-endpoint project; increases file navigation overhead.
- **Why rejected**: A single `Program.cs` is cleaner and conforms to minimal Web API style for simple apps.

### Alternative 3: Direct Method Testing

- **Description**: Test handlers by instantiating them directly as regular classes and calling their methods.
- **Pros**: Very fast unit tests; no HTTP server setup.
- **Cons**: Misses HTTP-level serialization, routing, validation, and status code behavior.
- **Why rejected**: WebApplicationFactory guarantees we test the full API layer, including model binding and standard RFC 7807 responses.

## Consequences

### Positive

- Zero setup overhead for databases.
- Clean, focused codebase with everything in `Program.cs`.
- High confidence in API behavior through real HTTP-level testing.
- Compliance with API standards (RFC 7807).

### Negative

- Pet data is lost on server restart.
- `Program.cs` will become crowded if many more endpoints are added in the future.

### Risks

- If the endpoint scale increases, we will eventually need to refactor out of a single file and introduce actual database storage. This risk is acceptable for this phase.
