# ADR-001: Product Approach for Pet Registration

## Status

Accepted

## Date

2026-06-02

## Context

We need to add a pet registration feature to the Web API. The system needs to support registering a pet's name, breed, date of birth, and tutor's name. We evaluated different product strategies regarding scope, user friction, and complexity of owner/tutor relations.

## Decision

We chose **Approach 1: Single-Step Standard Pet Registration**. 
This approach registers the pet and tutor details simultaneously in a single request. The tutor's name is collected as a simple text field directly within the pet registration resource. 

## Alternatives Considered

### Alternative 1: Phased Owner & Pet Profile Registration

- **Description**: A two-tier product model where tutor profile details (such as email/phone contacts) are collected alongside pet registration, laying the foundation for separate tutor entities and future notifications.
- **Pros**: Better preparation for future features like automated owner communication, grouping pets by tutor, and a richer data model.
- **Cons**: High initial complexity for clinic staff and clients, more verbose input requirement, and potential empty/sparse fields if contact details are missing.
- **Why rejected**: The primary goal is a simple, low-friction endpoint for rapid registration of pets in a veterinary/hospital context, and the added complexity of managing contact info/separate tutor profiles is out of scope for the initial MVP.

## Consequences

### Positive

- Simple, fast, and low-friction data entry for veterinary clinic users.
- Extremely rapid development and implementation of the registration endpoint.
- Easy to integrate for API clients.

### Negative

- Simple text representation of tutor names makes it difficult to model advanced parent-child relationship structures or group pets under a single unique owner record.
- Lack of secondary contact details (e.g. phone, email) limits communication features.

### Risks

- Migrating to a normalized owner/tutor profile model in the future will require API schema versioning/migration and data mapping.

## Implementation Notes

The endpoint should accept a request with:
- `petName` (required string)
- `breed` (required string, free text)
- `birthDate` (required date, must be in the past)
- `tutorName` (required string)
